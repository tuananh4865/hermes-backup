## Key Decisions

1. **How to inject recommendations into context** — Rather than modifying `_build_system_prompt` (7 overloads), appended recommendations to the user message as `injected_context` via `user_message["content"] = f"{_proactive_context}\n\n---\n\n{user_message['content']}"`.

2. **Pattern warning display** — Used `_emit_status` with `lifecycle:pre_execution` phase, consistent with existing Hermes status callback pattern. Shows recommendations if any, otherwise "No known failure patterns."

3. **`ProactivePlanResult` early return fix** — Added `adjusted_iteration_budget` and `abort_reason` to both early return paths in `plan_ahead()`.

4. **CRITICAL: `ComplexityAnalyzer.get_adjustment()` bug** — Was reading from `self._learned_adjustments` (in-memory dict) which is NEVER populated because: (a) `learn_from_trajectories()` is never called from `__init__`, and (b) even if it were, it reads `TrajectoryMatch` objects while `get_adjustment()` returns `dict`. Fixed by making `get_adjustment()` directly call `self._trajectory_index.get_adjustment()` when available, falling back to in-memory overrides.

5. **Boost application requires `medium_keywords >= 1`** — The `_classify_from_signals` method checks `if self._effective_high_kw > 1 and medium_keywords >= 1` before applying the boost. Tasks with 0 medium keywords never get boosted regardless of boost value.

6. **Indexed `task_type` must match analyzed task's classified type** — `get_adjustment(task_type)` uses the task_type from `_extract_signals` on the analyzed task's messages, NOT the `task_type` stored in indexed signals. `"Debug multiple files..."` → classified as `"bugfix"`, not `"file_editing"`. Test fixtures must align these.

7. **`correction_count=4` for `high_kw_boost`** — The `get_adjustment` logic: `if avg_corrections > 3` → `high_kw_boost`. `correction_count=0` → avg=0, no boost. `correction_count=4` → avg=4 > 3 → boost.
