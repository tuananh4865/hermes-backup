**Talk: "Context Engineering for Coding Agents" — Dex Hunter-Tarney**

Đây là talk từ AI Engineer Summit, về cách khiến AI coding agent hoạt động hiệu quả trong codebase thực tế (brownfield), tránh "slop" (code rác).

---

**Vấn đề gốc:**
- Khảo sát 100K developers → AI giúp ship nhiều hơn NHƯNG phần lớn là rework, đặc biệt với codebase cũ (10+ năm)
- Greenfield (dự án mới) → OK. Brownfield (codebase cũ) → nhiều slop

---

**4 nguyên tắc cốt lõi:**

**1. Context là tất cả**
- LLM không có "trí nhớ", chỉ dựa vào conversation context
- Điều duy nhất ảnh hưởng đến output là những tokens trong context
- "Garbage in, garbage out" — sai thông tin > thiếu thông tin > quá nhiều noise
- Warning: đừng để trajectory trở thành "I do something wrong, human yells at me, I do something wrong again"

**2. Tránh "Dumb Zone"**
- Context window có ~168K tokens
- Trên ~40% utilization → bắt đầu giảm hiệu quả
- Nhiều MCPs đang dump JSON UUIDs vào context → chết

**3. Intentional Compaction**
- Thay vì hỏi → sai → hỏi lại → sai → (lặp), dùng **sub-agents**
- Sub-agent đi tìm hiểu code → trả về kết quả ngắn gọn cho parent agent
- Frequent intentional compaction = giữ context luôn nhỏ

**4. Research → Plan → Implement (RPI)**
- **Research**: hiểu codebase thật sự (sub-agents đọc files, tìm relevant code)
- **Plan**: viết ra exact steps, file names, line numbers, cách test — nên include code snippets thật
- **Implement**: chạy theo plan

---

**Key insights:**

- **Đừng outsource thinking** — AI chỉ amplify thinking bạn đã làm hoặc đã KHÔNG làm
- **Mental alignment** quan trọng hơn code review thông thường — leader đọc plans để hiểu codebase đang thay đổi thế nào
- Mitchell (Boundary ML) để AMP threads trong PR để reviewer thấy journey thay vì chỉ green text
- Độ khó của task quyết định bao nhiêu "context engineering" cần thiết: đổi màu button → chỉ chat; simple feature → viết plan nhỏ; complex cross-repo → full RPI

## Design: Task Complexity Brain cho Hermes

### Vấn đề cần giải quyết

Hiện tại agent loop chạy tuần tự:
```
User message → LLM decide tool → execute → repeat
```
Không có **nhận biết trước** task lớn → không chủ động spawn research.

### Ý tưởng: Proactive Complexity Analyzer

Gắn vào **đầu mỗi turn**, trước khi LLM bắt đầu suy nghĩ:

```
run_conversation()
  └─→ complexity_check(user_message, conversation_history)
        ├─ Heuristics (fast path — no LLM call)
        └─ LLM classification (slow path — lightweight call)
      ├─ LOW    → proceed normally
      ├─ MEDIUM → spawn 1-2 research subagents, wait, inject findings
      └─ HIGH   → spawn parallel research subagents (codebase + tests + deps), 
                   wait, inject findings + suggest plan structure
```

### Hai layer nhận biết

**Layer 1: Heuristics (instant, no LLM call)**
- Keywords: `refactor`, `migrate`, `remove dependency`, `architect`, `design`, `benchmark`
- Multi-repo mention: `backend/`, `frontend/`, `shared/`
- File count hint: `change 10 files`, `across 3 modules`
- Stack overflow/error dump > 500 chars
- Task duration hint: `7 hours`, `ship 35k lines`

**Layer 2: LLM classifier (lightweight, 1 call)**
- Prompt: "Is this a complex coding task requiring codebase research? Answer: SIMPLE / MEDIUM / COMPLEX + reasoning in 2 sentences"
- Model: fast/cheap model được dùng
- Threshold: COMPLEX → trigger subagent spawning

### Spawning logic

```
COMPLEX task detected
  │
  ├─ Subagent #1: "Find relevant files for: [user goal]"
  │    Toolset: file, terminal (grep, find)
  │    Max iterations: 15
  │    Returns: list of relevant files with line numbers + brief description
  │
  ├─ Subagent #2: "Find related tests for the files found"
  │    Toolset: file, terminal
  │    Max iterations: 10
  │    Returns: test file paths + what they cover
  │
  └─ Subagent #3: "Find dependencies/imports for the core files"
       Toolset: terminal (grep imports)
       Max iterations: 5
       Returns: dependency graph summary

Parent waits (parallel) → collects results → inject as context block
→ THEN proceeds with normal LLM loop
```

```python
class PatternMatcher:
    """
    Given a new task, find similar past failures and extract
    lessons learned for proactive avoidance.
    """
    
    def match(self, task: str, signals: TaskSignals) -> PatternMatchResult:
        """
        Returns:
            - similar_failures: List[TrajectoryMatch]
            - failure_warnings: List[str]  ("Don't do X like last time")
            - success_hints: List[str]      ("Task Y succeeded with Z approach")
            - recommended_complexity: str   (upgrade/downgrade based on history)
        """
        
    def _compute_task_embedding(self, task: str) -> List[float]:
        """Simple TF-IDF or hash-based embedding for similarity."""
        
    def _should_warn(self, match: TrajectoryMatch) -> bool:
        """Only warn on high-similarity failures."""
        
    def _generate_warning(self, failure: TrajectoryMatch,
                          signals: TaskSignals) -> str:
        """Human-readable warning for the agent."""
```

### Phase 6: Learning Loop (Closed)

```
Task completes
  │
  ├─→ OutcomeEvaluator.classify()
  │     ├─ SUCCESS → index as success pattern
  │     └─ FAILURE → index with failure_reason + signals
  │
  └─→ Next task → PatternMatcher finds similar past
        └─→ warnings/hints injected into research context
             └─→ agent makes better decisions
```

**What gets learned:**

| Signal | What the system learns |
|---|---|
| `CONTEXT_OVERFLOW` at task type X | Future X → compact context early |
| `TRAJECTORY_CORRECTION_LOOP` > 3 corrections | Future X → spawn research subagent first |
| `ITERATION_EXHAUSTED` on medium tasks | Increase iteration budget for medium+ |
| `SLOP_GENERATED` on style-sensitive code | Include style guide in context |
| `MISSING_DEPENDENCY` on import tasks | Pre-scan dependencies before coding |
