# Apple Skills Repository Maintenance Guide

## Summary
This concept page outlines the structure, roles, and critical guidelines for maintaining the `apple-skills` GitHub repository. It serves as a maintainer's guide, detailing how to interact with human users, downstream coding agents, and automated tooling. The core focus is differentiating between hand-authored files and generated documentation to ensure efficient updates and accurate skill behavior.

## Key Insights
- **Role-Based Documentation:** The repository defines clear roles for different file types (`README.md` for users, `SKILL.md` for agents, `MAINTAINING.md` for maintainers).
- **Generated vs. Hand-Authored Rules:** A strict pattern-based system is in place to determine whether a file should be manually edited or regenerated by tooling, preventing manual edits on generated artifacts.
- **Maintenance Workflow Focus:** The primary workflow involves using dedicated scripts (`scripts/`), running checks (`package.json`), and leveraging the refresh tooling rather than direct file modification for generated content.

## Analysis
The `apple-skills` repository is structured not just as a collection of skills, but as an ecosystem designed for both human users and automated agents. The maintainer guide emphasizes that files have specific audiences: `README.md` targets end-users, while files like `skills/<name>/SKILL.md` are consumed directly by downstream coding agents. This separation ensures clarity regarding who is responsible for updating which part of the skill lifecycle.

A critical component of this repository structure is the distinction between hand-authored and generated content. The rules dictate that files matching specific patterns (e.g., `skills/**/*.md` with certain frontmatter) are considered generated, meaning maintainers should use refresh tooling rather than manual edits to update them. Conversely, root documentation (`README.md`, `CHANGELOG.md`) or configuration files (`package.json`, `tsconfig.json`) are treated as hand-authored and require direct maintenance.

The repository also establishes clear boundaries for maintainers, explicitly forbidding the conversion of user-facing documents into internal scripts and mandating that generated content fixes be applied at the source (the generator or source mapping), not by polishing the output artifact. This strict adherence to pattern-based ownership ensures that adding new skills or updating existing ones does not necessitate constant manual review of every file, promoting a scalable maintenance workflow.
