# Vector Databases

## Overview

A vector database is a specialized database system designed to store, index, and query high-dimensional vector embeddings — numerical representations of data (text, images, audio) generated by machine learning models. Unlike traditional relational databases that store exact values and query for equality, vector databases excel at semantic similarity search: finding items whose vector representations are close to each other in high-dimensional space, even when there's no lexical or keyword overlap.

The rise of vector databases is tightly coupled to the explosion of large language models and [[RAG]] (Retrieval-Augmented Generation) systems. LLMs are excellent at generating text but have limitations — they have knowledge cutoff dates, can hallucinate, and are expensive to fine-tune on specific domains. RAG solves this by retrieving relevant documents at query time and feeding them to the LLM as context. At the heart of RAG is the ability to represent documents and queries as vectors and find the most semantically similar ones — exactly what vector databases are built for.

## Key Concepts

**Embeddings:** Numerical vector representations of data generated by neural networks. Text embeddings map sentences or paragraphs to dense vectors (e.g., 1536 dimensions for OpenAI's `text-embedding-3-small`). Semantically similar texts produce vectors that are close together in the embedding space, enabling similarity search via distance metrics like cosine similarity or Euclidean distance.

**Approximate Nearest Neighbor (ANN) Search:** Exact nearest neighbor search doesn't scale to millions or billions of vectors. ANN algorithms (HNSW, IVF, PQ) trade off slight accuracy for massive speed improvements, enabling queries to return results in milliseconds even at web scale. HNSW (Hierarchical Navigable Small World) is the dominant algorithm in modern vector databases.

**Indexing:** Vector databases build special index structures to accelerate similarity searches. These indexes partition the vector space so that only a relevant subset of the space needs to be searched. The choice of index type significantly affects the trade-off between query speed, memory usage, and recall.

**Hybrid Search:** Combining vector similarity search with traditional keyword search (BM25). This is important because pure vector search can miss exact matches (e.g., product codes, names) that keyword search handles well. Most production RAG systems use hybrid approaches.

**Metadata Filtering:** In addition to semantic similarity, vector databases typically support pre-filtering or post-filtering by metadata (e.g., "only documents from 2024" or "only items in category 'electronics'"). This is essential for production applications where scope matters.

## How It Works

When a document is stored in a vector database:

1. **Chunking:** The document is split into chunks (sentences, paragraphs, or fixed token windows)
2. **Embedding:** Each chunk is passed through an embedding model to produce a dense vector
3. **Storage:** The vector, original text, and metadata are stored in the database
4. **Indexing:** An ANN index is built over the vectors for fast retrieval

At query time:

```python
# Pseudocode for vector similarity search in RAG
query_text = "What is the company's return policy?"
query_vector = embedding_model.encode(query_text)  # [1536-dim vector]

# Search for top 5 semantically similar chunks
results = vector_db.search(
    vector=query_vector,
    top_k=5,
    filter={"source": "policy_documents"}  # metadata filter
)

# Feed results as context to LLM
context = "\n".join([r.text for r in results])
response = llm.generate(f"Context: {context}\n\nQuestion: {query_text}")
```

This is the core RAG pattern — the vector database retrieves relevant context, and the LLM generates an answer grounded in that context.

## Practical Applications

- **RAG for LLMs:** Production-grade RAG systems use vector databases to ground LLM responses in organizational knowledge
- **Semantic search:** Next-generation search engines that understand query intent rather than just matching keywords
- **Recommendation systems:** Finding similar items (products, movies, songs) based on embedding similarity rather than collaborative filtering alone
- **Image similarity search:** Finding visually similar images in large collections (stock photo, fashion, e-commerce)
- **Anomaly detection:** Identifying outliers in embedding space as potential fraud or defects
- **Drug discovery:** Comparing molecular structure embeddings to identify promising compounds

## Examples

- **Pinecone:** Fully managed cloud vector database; popular for production RAG applications
- **Weaviate:** Open-source vector search engine with built-in modules for media types and hybrid search
- **Chroma:** Open-source, developer-friendly vector database optimized for AI applications; popular in prototyping and smaller deployments
- **Qdrant:** Open-source vector database with strong filtering capabilities and Rust implementation for performance
- **FAISS (Facebook AI Similarity Search):** Open-source library for dense vector similarity search; not a full database but widely used for embedding search in production systems
- **Milvus:** Open-source vector database built for scale; supports trillion-scale similarity search

## Personal Notes

Vector databases became a hot category seemingly overnight, but the underlying technology (ANN search) has been around for years. The real unlock was combining good vector search with good metadata filtering and a developer-friendly API — many early adopters struggled because pure vector search without metadata filters wasn't precise enough for production use cases. My takeaway: start with Chroma or Qdrant for prototyping; move to Pinecone or Weaviate for production. The choice between open-source and managed matters more as you scale. Also, chunking strategy is underrated — how you split documents dramatically affects retrieval quality.
