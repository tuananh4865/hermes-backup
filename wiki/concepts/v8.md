# V8

## Overview

V8 is Google's open-source JavaScript and WebAssembly engine, written in C++. It powers Google Chrome and Node.js, and is also used in other projects like Deno and Cloudflare Workers. V8's key innovation is its use of JIT (Just-In-Time) compilation to execute JavaScript—rather than interpreting bytecode slowly, V8 compiles JavaScript to native machine code at runtime, applying aggressive optimizations to achieve performance comparable to statically compiled languages.

V8 was first released in 2008 alongside Google Chrome and has continuously evolved. It introduced numerous performance techniques including hidden classes, inline caching, generational garbage collection (Orinoco and later Maglev), and Sparkplug—a non-optimizing JIT compiler designed for fast compilation startup. Understanding V8 helps developers write performant JavaScript and understand why certain coding patterns are faster than others.

## Key Concepts

**JIT Compilation**: V8 compiles JavaScript to native machine code on the fly. The Ignition interpreter first profiles execution, then the TurboFan optimizing compiler recompiles hot functions with type-specialized optimizations.

**Hidden Classes**: Despite JavaScript's dynamic typing, V8 creates internal hidden classes (shapes) for objects with the same property structure, enabling efficient property access similar to class-based languages.

**Inline Caching**: V8 caches the results of property lookups and function calls at call sites, avoiding expensive runtime type checks on repeated execution.

**Garbage Collection**: V8 uses generational garbage collection with Orinoco (full collection) and Oilpan (compacting) for large objects. Young generation collection (scavenge) is frequent and fast.

**TurboFan Optimizing Compiler**: The advanced JIT compiler performs speculative optimizations based on observed types, using techniques like deoptimization when assumptions are violated.

## How It Works

1. **Parsing**: JavaScript source is parsed into an AST (Abstract Syntax Tree).
2. **Ignition**: The interpreter compiles the AST to bytecode and begins execution, collecting type feedback (shapes, call frequencies).
3. **Sparkplug**: A simple, fast non-optimizing compiler compiles frequently called functions immediately after Ignition for faster startup.
4. **TurboFan**: When a function becomes hot (called many times), TurboFan recompiles with optimizations like inlining, loop unrolling, and type specialization.
5. **Deoptimization**: If TurboFan's type assumptions fail, it deoptimizes back to bytecode, ensuring correctness at the cost of performance.

```text
JavaScript Source Code
    │
    ▼
Parser → Abstract Syntax Tree (AST)
    │
    ▼
Ignition Interpreter
    │        \
    │         └──► Bytecode + Feedback (shapes, counts)
    │                      │
    │                      ▼
    │              Sparkplug Compiler
    │              (fast, non-optimizing)
    │                      │
    ▼                      ▼
    │              Hot Functions Detected
    │                      │
    └──────────────────────┴──► TurboFan Optimizing Compiler
                                    │
                                    ▼
                              Native Machine Code
```

## Practical Applications

- **Web Browsers**: Running client-side JavaScript in Chrome with high performance
- **Server-side Runtime**: Powering Node.js for building web servers, CLI tools, and backend services
- **Edge Computing**: Deploying JavaScript to edge locations (Cloudflare Workers) for low-latency responses
- **Desktop Applications**: Building desktop apps with Electron using V8 for rendering and logic
- **Serverless Functions**: Executing short-lived JavaScript functions in serverless environments

## Examples

Inspecting V8's optimization decisions:

```javascript
// Run this in Chrome DevTools or Node.js with --trace-opt flag

function add(a, b) {
  return a + b;
}

// This function will be optimized by TurboFan
// once V8 collects enough type feedback
for (let i = 0; i < 100000; i++) {
  add(i, i + 1);
}

// Check optimization status in Node.js
// node --trace-opt --trace-deopt test.js
```

Using V8's heap statistics:

```javascript
// In Node.js
const v8 = require('v8');

console.log(v8.getHeapStatistics());
// {
//   total_heap_size: 150081536,
//   total_heap_size_executable: 4194304,
//   total_physical_size: 113864704,
//   used_heap_size: 98594304,
//   heap_size_limit: 2197815296,
//   ...
// }

// Memory statistics by space
console.log(v8.getHeapSpaceStatistics());
```

## Personal Notes

Understanding V8's optimization heuristics has changed how I write JavaScript. Consistent object shapes (no dynamic property addition), avoiding try-catch in hot paths, and favoring monomorphic calls over polymorphic ones all make a measurable difference. I use `--trace-opt` and `--trace-deopt` flags regularly when profiling performance-critical code. The deoptimization bailout messages are cryptic but invaluable when TurboFan can't optimize a function as expected.
